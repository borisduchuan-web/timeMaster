# AI APP Codex 多智能体全链路开发计划

版本：v0.1  
日期：2026-05-28  
开发模式：Codex 总控 + 多专项 Agent 并行开发  
目标平台：Android App + Java Spring Boot 后端 + Web 后台 + 自动化测试

---

# 一、开发模式定义

本项目不采用传统“人工逐行编码”的开发模式，而采用：

```text
人类创始人/产品负责人
→ Codex 总控 Agent
→ 多专项开发 Agent
→ 自动化测试 Agent
→ 总控集成、验收、修复、发布
```

人类主要负责：

* 方向决策
* 需求确认
* 版本取舍
* 验收反馈
* 商业/运营/法务判断

Codex 总控负责：

* 拆分需求
* 分派 Agent
* 约定接口
* 控制代码边界
* 整合代码
* 处理冲突
* 组织测试
* 输出版本

专项 Agent 负责：

* 在明确边界内独立开发
* 提交可运行代码
* 自测并说明风险
* 不覆盖其他 Agent 的工作

# 二、Agent 角色编组

## 1. 总控 Agent

角色：项目经理 + 架构师 + 集成负责人

职责：

* 拆分版本目标
* 分配任务给专项 Agent
* 维护接口契约
* 审查关键代码
* 合并前后端成果
* 处理跨模块问题
* 控制版本范围
* 输出每日进度和发布结论

总控不把所有代码都交出去，而是保留：

* 架构决策

# 四、版本目标

## 1. 7 天原型版

目标：

证明核心闭环能跑。

必须跑通：

```text
注册登录
→ 查看任务
→ 查看任务流程节点
→ 接单
→ 执行节点
→ 上传图片/短视频证据
→ 提交任务
→ 后台验收
→ 成长值变化
```

允许：

* UI 粗糙
* 部分数据 Mock
* 本地 AI 评分占位
* 无真实支付
* 无复杂风控

不允许：

* 主流程跑不通
* 任务重复接单
* 状态乱跳
* 证据上传后找不到
* 成长重复结算

## 2. 14 天 MVP 版

目标：

形成可稳定演示的产品雏形。

必须具备：

* 完整任务状态机
* 灰显节点
* 流程模板
* 证据包
* 后台验收/驳回/返工
* 成长体系
* 技能熟练度
* 声望流水
* 个人知识库
* 虚拟钱包流水
* API 自动化测试
* Android APK

验收标准：

```text
P0 缺陷 = 0
状态机合法/非法迁移测试通过
重复请求不重复结算
证据链可追溯
成长流水可追溯到任务和节点
```

## 3. 30 天试运营版

目标：

小范围真实用户可试用。

必须具备：

* 子单机制
* AI 接管状态预留
* 争议/返工/取消流程
* SLA 超时处理
* 文件上传重试
* 视频压缩或上传优化
* 弱网恢复
* 操作审计
* 风控冻结
* 简单数据看板
* 崩溃日志和后端日志
* Docker Compose 一键启动

验收标准：

```text
P0 = 0
P1 = 0
核心 API 自动化覆盖率 >= 85%
状态机测试覆盖率 = 100%
证据上传成功率 >= 95%
安卓核心流程可在多台真机通过
```

# 八、Agent 协作规则

## 1. 文件边界

每个 Agent 只能修改自己的目录：

```text
后端 Agent：backend/
Android Agent：android/
后台 Agent：admin-web/
测试 Agent：tests/ 和对应测试目录
DevOps Agent：infra/、docker-compose.yml
文档 Agent：docs/
```

跨目录修改必须由总控 Agent 处理。

## 2. 接口契约优先

前后端不能各写各的。

所有接口先进入：

```text
docs/api-contract.md
```

再分别实现。

## 3. 状态机统一

状态机只能由后端统一实现。

Android 和后台只展示状态，不自行推导最终状态。

## 4. 测试先行

状态机、接单、结算、成长必须先写测试。

没有测试的核心逻辑不能进入 MVP。

## 5. 每日集成

每天至少集成一次。

每日输出：

```text
今日完成
明日目标
阻塞问题
P0/P1 缺陷
可演示链路
```

# 十、真实周期判断

在多 Agent 并行、需求范围不频繁变化的前提下：

| 版本 | 时间 | 性质 |
|---|---:|---|
| 可点击假 Demo | 1-3 天 | 主要展示 UI 和概念 |
| 可运行原型 | 7 天 | 真实前后端闭环，稳定性有限 |
| 可演示 MVP | 14 天 | 核心状态机、证据链、成长体系可信 |
| 小范围试运营 | 30 天 | 可让少量真实用户试用 |
| 商业上线 | 2-4 个月 | 需要支付、法务、客服、风控、安全、运营体系 |

建议当前目标：

```text
14 天做出可演示 MVP
30 天进入小范围试运营
```

---

# 十一、第一版坚决不做

为了保证多 Agent 开发不失控，第一版不做：

* 真实支付
* 复杂端侧 AI 视频模型
* 完整自动派单算法
* 多城市调度
* 商业保险
* 复杂仲裁系统
* 大规模权限后台
* 机器人执行数据闭环

第一版只预留字段和接口，不进入复杂实现。

# 十二、总控结论

这个项目可以交给 Codex 多智能体链路执行，但不能让 Agent 随意发挥。

必须采用：

```text
总控锁范围
后端锁状态机
前端锁核心流程
测试锁验收标准
后台锁运营闭环
DevOps 锁可运行环境
```

第一阶段最重要的不是页面数量，而是把这条链打通：

```text
任务流程节点
→ 灰显/启用
→ 接单执行
→ 证据采集
→ 后台验收
→ 成长结算
→ 知识库沉淀
```

这条链跑通后，平台就已经具备区别于普通接单软件的核心产品形态。
